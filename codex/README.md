# Codex MCP Stack

Дата: 2026-03-19

## Назначение

Этот каталог описывает MCP-стек для `Codex app` на этой машине:

- какие серверы подключены глобально;
- зачем каждый нужен;
- какие у него ограничения;
- как они дополняют друг друга.

Канонический подробный документ:

- `mcp.md`

Источник истины по факту подключения:

- `~/.codex/config.toml` для custom MCP;
- `docker mcp server ls` для серверов, включенных внутри `MCP_DOCKER`.

## Логика слоями

Стек разбит на три слоя:

1. `Custom servers` в `Codex app`
   Это постоянные, понятные серверы под реальные ежедневные сценарии.
2. `MCP_DOCKER`
   Это gateway в Docker MCP catalog. Он нужен как универсальный шлюз к дополнительным MCP без захардкоживания каждого из них в `config.toml`.
3. Узкоспециализированный remote MCP
   Для работы с отдельной удаленной машиной используется `ssh-laptop`.

Практическое правило:

- если инструмент нужен часто и под твои реальные данные, лучше держать его отдельным custom server;
- если инструмент нужен эпизодически, можно оставлять его за `MCP_DOCKER`;
- если инструмент работает с конкретной машиной или конкретной базой, это надо фиксировать в названии и описании.

## Подключенные custom servers

### `MCP_DOCKER`

Роль:

- единая точка входа в Docker MCP catalog;
- позволяет искать, добавлять и конфигурировать дополнительные MCP через Docker toolkit.

Что дает:

- доступ к каталогу MCP;
- возможность быстро включать новые серверы без ручной сборки;
- контейнерную изоляцию для сторонних MCP.

Ограничения:

- сам по себе это gateway, а не предметный сервер под задачу;
- UI `Codex app` показывает его как один сервер, а не как список всех catalog-серверов внутри.

### `ssh-laptop`

Роль:

- удаленное выполнение команд на отдельной машине по SSH.

Что дает:

- доступ к серверным командам;
- проверку runtime, сервисов, логов, docker и systemd на удаленном хосте;
- полезен для `airsrv` и операционных задач.

Ограничения:

- это не файловый индексатор и не git-интеграция;
- он хорош для команд и диагностики, а не для локальной навигации по проектам.

### `notes-fs`

Роль:

- быстрый файловый MCP для всего дерева `/Users/doc/notes`.

Что дает:

- чтение файлов и каталогов;
- поиск по именам и содержимому;
- быстрый обзор больших деревьев;
- точечную работу с markdown, yaml, json, исходниками и медиа.

Текущая политика:

- область доступа ограничена `/Users/doc/notes`;
- запись включена через `ALLOW_WRITE=true`.

Когда использовать:

- для `train`, `monty`, `mediadistri`, `airsrv`, `todo`, `dd`;
- когда нужен файловый контекст без лишнего доступа ко всей машине.

### `notes-git`

Роль:

- git-операции по репозиториям внутри `/Users/doc/notes`.

Что дает:

- `status`, `log`, `diff`, `show`, `add`, `commit`, работа с ветками;
- годится для локальных репозиториев в `notes`, где важно быстро смотреть состояние без shell.

Ограничения:

- работает только с репозиториями, доступными из смонтированного дерева `/Users/doc/notes`;
- не заменяет GitHub API и не знает про PR, issues и remote metadata.

### `context7`

Роль:

- актуальная документация по библиотекам и пакетам.

Что дает:

- поиск library ID;
- загрузку свежей документации по темам;
- особенно полезен для `FastAPI`, `aiogram`, `SQLAlchemy`, `React`, `Vite`, `TypeScript`.

Когда использовать:

- когда нужен именно внешний API/framework context;
- когда локального кода недостаточно и не хочется идти в web search.

### `atlas-docs`

Роль:

- второй docs-MCP с навигацией по наборам документации.

Что дает:

- список доступных документационных наборов;
- поиск страниц;
- чтение отдельных страниц или полного набора.

Логика подключения:

- `context7` удобен для package-first поиска;
- `atlas-docs` удобен как docs index/search слой.

### `myshows-db`

Роль:

- отдельный database MCP, сразу привязанный к локальной SQLite-базе `series`.

База:

- `/Users/doc/notes/series/data/myshows_watch_history.db`

Что дает:

- просмотр схемы и таблиц;
- SQL-запросы;
- natural-language запросы к базе;
- быстрый анализ watch history без ручного запуска sqlite CLI.

Когда использовать:

- для аналитики по сериалам;
- для ad-hoc вопросов к базе без написания временных скриптов.

Ограничения:

- сервер завязан на одну конкретную базу;
- для других БД лучше поднимать отдельный именованный MCP, а не переиспользовать этот.

## Что сейчас живет внутри Docker MCP

На 2026-03-19 внутри `MCP_DOCKER` дополнительно включен:

- `playwright`

Его логика:

- это браузерная автоматизация и UI smoke/regression слой;
- особенно полезен для локальных интерфейсов `monty/dashboard`, `train/dashboard`, `train-service/dashboard`, `dd`.

Отдельно этот же сервер также подключен как custom server `playwright`, чтобы он был виден в UI `Codex app` отдельной строкой.

## Как этот стек использовать

Нормальная последовательность такая:

1. Локальный код и данные: `notes-fs`
2. История изменений и состояние репо: `notes-git`
3. Внешняя docs-справка: `context7` или `atlas-docs`
4. Специальные данные: `myshows-db`
5. Удаленный runtime: `ssh-laptop`
6. Редкие/новые интеграции: через `MCP_DOCKER`

## Что не подключено отдельным custom server

Отдельно не вынесены:

- `GitHub`
- `Docker CLI` как самостоятельный custom server
- общий `database-server`

Причины:

- `GitHub` потребует токен и отдельный auth-flow;
- `Docker CLI` пока логичнее держать через existing tooling и `MCP_DOCKER`, чем плодить дубли;
- общий database MCP имеет смысл добавлять только когда появится второй устойчивый database target.

## Правило именования

Имена custom servers должны быть:

- короткими;
- предметными;
- отражать scope.

Примеры:

- `notes-fs` — файловый доступ к `notes`;
- `notes-git` — git по `notes`;
- `myshows-db` — база только под один data source.

## Изменение конфигурации

После добавления или удаления custom MCP:

- проверить `codex mcp list`;
- проверить `~/.codex/config.toml`;
- полностью перезапустить `Codex app`, если UI не подхватил изменения сразу.

Это важно, потому что desktop UI может не перечитывать `config.toml` на лету.
